# Decision Record: standardized Data Quality Validation with Great Expectations

## 1. Context & Problem Statement

In an **Agentic MLOps** pipeline, we combine deterministic data (hard signals like churn metrics) with probabilistic data (soft signals generated by LLMs like ticket notes). If the "hard signals" drift or the "soft signals" fail to adhere to expected formats, the downstream ML models and AI Agents will perform poorly or crash unexpectedly.

**The Problem:**
*   **Silent Failures:** Data schema changes can propagate through the pipeline without erroring, but degrading model performance.
*   **Garbage In, Garbage Out:** Brittle `if/else` or `assert` checks are hard to maintain, don't generate reports, and don't scale across multiple pipeline stages.

---

## 2. The Decision: Great Expectations (GX) v1.0+

We have selected **Great Expectations (GX)** as the primary "Data Inspector" for this project.

### Why Great Expectations?

1.  **Expectations-as-Code:** It provides a standardized domain-specific language (DSL) to describe what data *should* look like (e.g., `expect_column_values_to_be_between`).
2.  **Separation of Concerns:** It acts as the **Inspector** Layer (Rule 2.1), checking data integrity without mutating it. This separates validation logic from the transformation logic (The Mechanic).
3.  **Rich Validation Metadata:** Unlike simple assertions, GX generates `ValidationResult` objects that contain specific details about which rows failed and why, enabling **Agentic Healing** (where an agent can read the error and try to fix the data).
4.  **Pipeline Guardrails:** By integrating GX into DVC stages, we ensure that a pipeline stage **stops immediately** if data quality conditions are not met, protecting downstream training/inference.

---

## 3. Implementation Logic: How it Works Here

The system implements the `DataValidator` component (`src/components/data_validation.py`) using the following patterns:

### 3.1 Ephemeral Context Strategy
We use `gx.get_context(mode="ephemeral")`.
*   **Why:** Rather than maintaining a complex local metadata store (the traditional `great_expectations/` folder), we keep the validation logic entirely in Python code. This makes the project more portable and easier to version-track.

### 3.2 Expectation Suites
We define two primary suites:
*   **Raw Telco Suite:** Validates the initial ingestion (null checks, tenure >= 0, categorical consistency).
*   **Enriched Telco Suite:** Validates the AI Agent's output (ensuring `primary_sentiment_tag` is valid and `ticket_notes` meet minimum length requirements).

### 3.3 The "Fail-Loud" Pattern
When a validation fails, the component:
1.  Writes a `status.txt` with `FAILED`.
2.  Generates a detailed `validation_report.json`.
3.  Raises a custom `StatisticalContractViolation` exception.
*   **Purpose:** This forces the DVC pipeline to stop and alerts the developer (or the Agent) with precise metadata about the violation.

---

## 4. Alternatives Considered

| Alternative | Why we didn't choose it |
|---|---|
| **Pandas Schema / Pandera** | Excellent for typing, but lacks the rich "Expectation" DSL and the automated reporting/metadata features of GX. |
| **Manual Assert Checks** | Hard to maintain, zero observability, and lacks standardization across different MLOps teams. |
| **Pydantic (for whole files)** | Pydantic is used for **Row-Level** contracts (our `TelcoCustomerRow`). GX is used for **Batch-Level** statistical assertions (e.g., "column X must be unique" or "column Y mean must be between 10-50"). |

---

## 5. Summary of Benefits
*   **Observability:** Every pipeline run produces a JSON report of data health.
*   **Reliability:** Guarantees that the "Training Pipeline" never consumes corrupted data.
*   **Agent-Ready:** Provides the structured feedback needed for multi-agent autonomous error correction.
