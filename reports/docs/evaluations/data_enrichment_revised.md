## Revised Implementation Plan

### Root Cause (Confirmed)

The leakage operates on two levels simultaneously, both requiring surgical fixes:

**Level 1 — `CustomerInputContext` schema** (`src/components/data_enrichment/schemas.py`): The `Churn: Literal["Yes", "No"]` field is passed directly to the LLM. The model uses it to write notes that explicitly reference churn intent ("planning to switch", "decision to churn") in 98.3% of churner rows.

**Level 2 — System prompt** (`src/components/data_enrichment/prompts.py`): Even without the explicit `Churn` field, the current prompt instructs the LLM to generate a note that reflects the customer's outcome. The prompt must be rewritten to adopt the perspective of a **support agent writing notes during a live call** — before any churn decision is observable.

**Level 3 — `primary_sentiment_tag`** (confirmed, Decision A2 was correct): The tag is 98.3% `Frustrated` for churners and 93.1% `Satisfied` for non-churners. Excluding it from training (Decision A2) was the right call and remains unchanged. However, since the tag is generated by the same leaky prompt, it too will become a more honest signal after regeneration — still excluded from training but more realistic for diagnostic use.

### Legitimate Grounding Signals for the New Prompt

The new prompt receives all 19 structured features **except `Churn`**. These are the signals a real support agent would see in the CRM during a call:

| Signal | How it legitimately grounds the note |
|---|---|
| `Contract` | Month-to-month → customer may be evaluating options; Two year → committed |
| `tenure` | Short tenure (1–3 months) → onboarding friction; Long tenure → loyalty context |
| `MonthlyCharges` | High charges → billing inquiry likelihood |
| `InternetService` + `TechSupport=No` | Fiber with no support → legitimate complaint driver |
| `OnlineSecurity`, `OnlineBackup` | Absent services → reasonable source of concern |
| `SeniorCitizen` | May ground the tone and complexity of the note |
| `PaymentMethod` | Electronic check → often correlated with disengagement |

The new prompt instructs: *"Write a 1–3 sentence support agent note summarising this customer's interaction. Base it only on the service profile visible in the CRM. Do not reference whether the customer has churned or will churn."*

### Exact Leakage Points Identified

**`src\components\data_enrichment\schemas.py` — 1 field removal:**
`Churn: Literal["Yes", "No"]` in `CustomerInputContext`. This is the primary leakage vector — it feeds the label directly into the LLM prompt.

**`src\components\data_enrichment\generator.py` — 2 changes:**
1. `user_prompt` construction at line 54–61 explicitly injects `Churn Status: {customer_context.Churn}` into the formatted prompt string. This must be removed.
2. The **deterministic fallback** (Phase 3, lines 91–108) uses `customer_context.Churn` to branch between "Frustrated" and "Satisfied" outcomes. This is leakage in the last-resort path and must be replaced with feature-based logic.

**`src\components\data_enrichment\prompts.py` — full system prompt rewrite:**
The current `LOGIC GATES` section explicitly instructs the LLM with rules like *"High Risk / Churned (... AND Churn=Yes) → Generate severe complaints"* and *"If `Churn=Yes`, the note MUST be negative."* These instructions hard-code the label into the LLM's reasoning chain.

**`src\components\data_enrichment\orchestrator.py` — 1 field removal:**
`_process_row()` at line 52 passes `Churn=str(row.get("Churn", "No"))` into `CustomerInputContext`. Once `Churn` is removed from the schema, this argument must be removed from the constructor call.

**`tests\unit\test_enrichment.py` — 3 test updates:**
All three tests that construct `CustomerInputContext` pass `"Churn": "Yes"` or `"Churn": "Maybe"`. These must be updated to reflect the new schema. The `test_customer_input_context_invalid_literals` test currently relies on `Churn: "Maybe"` as one of its two invalid fields — it needs a replacement invalid field to preserve its intent.

**`src\components\stage_02_data_enrichment.py` and `src\components\stage_03_enriched_validation.py` — no changes required.** Both are conductors with no direct reference to the `Churn` field.

### New `CustomerInputContext` — Field Expansion

The schema gains fields to compensate for losing the label — giving the LLM more legitimate CRM signals to ground a realistic note. This matches what a real support agent would actually see:

```
Removed : Churn
Added   : gender, SeniorCitizen, Partner, Dependents, PhoneService,
          MultipleLines, OnlineSecurity, OnlineBackup, DeviceProtection,
          StreamingTV, StreamingMovies, PaperlessBilling, PaymentMethod
```

Retained from original: `customerID`, `tenure`, `InternetService`, `Contract`, `MonthlyCharges`, `TechSupport`.

The expanded context ensures the LLM has enough signal to write contextually rich notes without needing the label. A customer with Fiber optic + No TechSupport + No OnlineSecurity + month-to-month + high charges will legitimately receive a frustrated note from observable signals alone.

### New Deterministic Fallback Logic

The fallback in `src\components\data_enrichment\generator.py` must be rewritten using only observable features:

```python
# High-friction profile: fiber + no support + month-to-month
if (customer_context.InternetService == "Fiber optic"
        and customer_context.TechSupport == "No"
        and customer_context.Contract == "Month-to-month"):
    note = (f"Customer contacted support regarding recurring connectivity issues "
            f"with their Fiber optic service. Monthly charges of "
            f"${customer_context.MonthlyCharges:.2f} were flagged as a concern.")
    sentiment = "Frustrated"

# Price-sensitive: high charges + month-to-month
elif (customer_context.MonthlyCharges > 70
        and customer_context.Contract == "Month-to-month"):
    note = (f"Customer called to inquire about billing after noticing their "
            f"monthly charges of ${customer_context.MonthlyCharges:.2f}. "
            f"Requested information on available promotions.")
    sentiment = "Billing Inquiry"

# Loyal: long-term contract
elif customer_context.Contract == "Two year":
    note = (f"Customer on a two-year contract inquired about available "
            f"service upgrades. Expressed general satisfaction with current plan.")
    sentiment = "Satisfied"

# Default: neutral inquiry
else:
    note = (f"Customer with {customer_context.tenure} months tenure called "
            f"regarding their {customer_context.InternetService} service plan.")
    sentiment = "Neutral"
```
